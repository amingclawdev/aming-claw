## Design Background

The governance chain (PMâ†’Devâ†’Testâ†’QAâ†’Merge) flows fully automatically by default: as soon as a task is created, the executor claims it immediately, and as soon as a gate passes, the next stage is triggered right away â€” with no human review window at any point.

The design goal of Observer Mode is: **to insert review checkpoints for an observer (human or Claude) without breaking the governance chain structure**.

---

## Core Concepts

### New Task Status: `observer_hold`

```
queued â†’ observer_hold    (hold_task API)
observer_hold â†’ queued    (release_task API)
observer_hold â†’ claimed   (observer takes over directly)
```

Semantics of `observer_hold`:
- The executor's claim skips this status (`WHERE execution_status IN ('queued')`)
- auto-chain gates do not trigger progression
- Only an explicit observer `release` or `claim` can break the pause

### Project-Level Switch: `observer_mode`

Stored in `project_version.observer_mode` (new column added in DB migration v9).

When `observer_mode = true`, **all newly created tasks** (including next-stage tasks generated by auto-chain) start in `observer_hold` status rather than `queued`.

## API Reference

### Toggle observer_mode

```bash
# Enable
POST /api/project/{pid}/observer-mode
{"enabled": true}

# Disable
POST /api/project/{pid}/observer-mode
{"enabled": false}

# Query
GET /api/project/{pid}/observer-mode
```

### Single-task hold / release

```bash
# Pause (queued â†’ observer_hold)
POST /api/task/{pid}/hold
{"task_id": "task-xxx"}

# Release (observer_hold â†’ queued)
POST /api/task/{pid}/release
{"task_id": "task-xxx"}
```

## Typical Use Cases

### Scenario A: Full observer takeover (human reviews every step)

```
1. observer_mode ON
2. Create coordinator task â†’ automatically enters observer_hold
3. Observer inspects (checks memory + queue + context)
4. User reviews â†’ task_claim(observer) â†’ executes coordinator logic
5. PM task produced â†’ also enters observer_hold
6. User reviews PM spec â†’ task_claim(observer) â†’ refines PM output
7. ... repeat for each stage ...
8. observer_mode OFF â†’ subsequent stages resume automatic flow
```

### Scenario B: Selective interception (only review specific stages)

```
1. observer_mode OFF (automatic by default)
2. Notice a queued task needs review â†’ task_hold(task_id)
3. Review complete â†’ task_release(task_id) or claim directly
4. Subsequent stages advance automatically
```

### Scenario C: Debug auto-chain (without affecting the executor)

```
1. executor_scale(0) (stop executor)
2. observer_mode ON
3. Manually trigger auto-chain step by step, observe each stage's output
4. executor_scale(1) (resume)
```

## Design Constraints

1. **`observer_hold` is not a terminal status**: Tasks will not time out, will not be reclaimed, and will not trigger retries.
2. **Normal auto-chain triggers after observer claim completes**: When completing a task, pass `completed_by="observer-xxx"` â€” auto-chain recognizes the observer identity.
3. **`observer_mode` applies to all tasks**, including test/qa/merge tasks generated internally by auto-chain. If you only need to intercept a specific stage, use `task_hold` to handle it individually.
4. **Executor can claim immediately after release**: If the executor is running (scale > 0), the task will be auto-claimed quickly after release. If the observer needs to execute personally, use `task_claim` instead of `task_release`.
